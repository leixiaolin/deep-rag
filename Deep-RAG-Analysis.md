# Deep RAG 项目分析

> 本文档对 `deep-rag` 项目的工作原理、优缺点以及可改进之处进行系统梳理。

---

## 一、工作原理

Deep RAG 与传统 RAG 的根本区别：**抛弃向量数据库和 embedding 相似度检索**，改用「LLM 驱动的主动检索」。

### 核心机制分两阶段

#### ① 离线构建「知识地图」（`Knowledge-Base-File-Summary/generate.py`）

- 递归扫描 `Knowledge-Base/` 下所有 `.md` 文件
- 小文件（<3000 tokens）直接让 LLM 生成摘要
- 大文件（>3000 tokens）先让 LLM 按语义切分（保留行号），再为每段生成摘要
- 最终汇总成 `summary.txt`——一棵带摘要的目录树，例如：

```
Product-Line-A-Smartwatch-Series/
├─ SW-2100-Flagship.md: 2.1" AMOLED, 72hr, IP68, $2999
└─ SW-1300-Youth.md: 1.3" OLED, 24hr, IP65, $599
```

#### ② 运行时主动检索（`backend/main.py:134-245`）

1. 把整个 `summary.txt` 注入 system prompt（`prompts.py:18-21`）
2. LLM 看到「地图」后，通过 `retrieve_files` 工具主动声明要读哪些文件/目录
3. 后端返回**完整文件内容**（非碎片化 chunk），追加到对话
4. 进入下一轮——LLM 可继续调用工具或给出最终答案
5. 最多循环 10 次（`main.py:171`）

### 两种工具调用模式（`config.py`）

- `function`：标准 Function Calling（主模式）
- `react`：用 `<|Thought|>/<|Action|>/<|Observation|>` 文本协议，兼容不支持函数调用的模型（`react_handler.py`）

### 端到端流程示例

**问题**：「全部地区去年总营收？」

LLM 看地图 → 调用 `retrieve_files(["2023-Market-Layout/"])` → 拿到 4 个区域完整文件 → LLM 抽取数值并求和 → 输出答案。

---

## 二、优点

| 优点 | 说明 |
|---|---|
| **架构简单** | 无需向量库、无需 embedding 模型，部署成本低 |
| **保完整上下文** | 返回整份文件而非 top-k 碎片，避免语义割裂 |
| **复杂查询能力强** | 否定查询（"除了 AMOLED 还有什么"）、数值比较、最值查找、多跳聚合——传统切块 RAG 的痛点都能很好解决 |
| **多模型兼容** | OpenAI / Gemini / Anthropic / 任意 OpenAI 兼容 API；不支持函数调用时自动降级 ReAct |
| **流式体验好** | SSE 流式返回内容、工具调用、工具结果三类事件，前端实时展示检索过程 |
| **教学价值高** | 代码量小（后端核心约 700 行），逻辑清晰，是理解 Agentic RAG 的优秀范例 |

---

## 三、缺点

### 1. 可扩展性瓶颈（最严重）

`summary.txt` 整体注入 system prompt——知识库一变大，prompt 就撑爆上下文窗口。当前 demo 只有约 43 个文件尚可，上千文件时方案直接失效。

### 2. 性能与成本

- 每次问答需多次 LLM 往返（检索 + 推理），延迟显著高于向量检索
- 每轮都把已检索文件追加进对话，token 消耗随迭代次数线性增长
- 无检索结果缓存，重复问题重复读文件

### 3. 检索完全依赖 LLM 判断

LLM 看摘要决定读哪个文件——**一旦误判就无任何兜底**。没有 embedding 相似度、关键词倒排索引作为 fallback。摘要质量直接决定召回率。

### 4. 摘要 prompt 过于苛刻

`generate.py` 中 `MAX_SUMMARY_TOKENS = 10`——10 token 的摘要很难承载足够信息，关键参数（价格、型号、数值）容易被丢弃，反过来又伤害检索精度。

### 5. 工程细节问题

- 目录检索时 `knowledge_base.py` 用 `rglob` 顺序读取，无并发
- `main.py:196-210` 工具调用累积逻辑只支持单工具/轮，且对 `index` 处理较粗糙
- `prompts.py:130` ReAct 的 `action_input` 正则 `\{[^}]+\}` 不支持嵌套对象，容错弱
- `main.py:87-89` 修改 `.env` 后用 `global settings` 重新实例化——但已注入到旧 `settings` 的模块引用不会更新
- 无重排（rerank）、无混合检索、无召回评估
- `summary.txt` 全量重新生成代价高，无增量更新机制

### 6. 安全性

- `main.py:67-93` 的 `/api/config` POST 端点直接写入 `.env` 文件，无鉴权——生产环境是严重漏洞
- CORS 配置为 `allow_origins=["*"]`

## 四、改进建议（按优先级）

### 高优先级

1. **分层摘要 / 摘要压缩**：目录级摘要 → 子目录摘要 → 文件摘要，按需展开，突破上下文窗口限制
2. **混合检索兜底**：保留知识地图作为主路径，额外建一份 embedding 索引作为召回兜底，防止 LLM 误判
3. **检索缓存层**：对已读文件内容做 LRU 缓存；对 (query → file_paths) 做语义缓存
4. **摘要 token 放宽**：10 → 30~50，并在 prompt 中强制要求保留数值、型号、专有名词
5. **`/api/config` 加鉴权**：至少加 token 校验或限制为 localhost

### 中优先级

6. **并发文件读取**：目录检索用 `asyncio.gather` 并行
7. **增量摘要更新**：按文件 mtime 哈希做差量，只重生成变动部分
8. **重排机制**：多文件召回后用小模型 rerank，再喂给主 LLM
9. **工具调用鲁棒性**：支持单轮多工具、修复正则、加 JSON schema 校验
10. **检索自评**：LLM 对召回结果打分，不满意自动重检索

### 低优先级

11. 支持非 markdown 文件（PDF / Word）
12. 加观测：检索命中率、迭代次数、token 消耗的 metrics
13. 前端增加「展开查看检索到的原文」交互

---

## 五、一句话总结

Deep RAG 用「**知识地图 + LLM 主动检索**」的 Agentic 思路，巧妙回避了向量库的复杂度，在中小规模结构化知识库上对复杂查询表现出色；但 **可扩展性、延迟、成本、检索兜底** 四大短板决定了它更适合做原型 / 演示，要上生产需补上分层摘要、混合检索和缓存层。
